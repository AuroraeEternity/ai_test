# 测试用例生成最终流程

---

## 一、目标定位

这份流程用于指导当前 AI 测试平台的后续演进，目标不是让 LLM 一次性直接生成全部用例，而是通过分阶段收敛上下文，持续提升测试用例的准确性、覆盖度和可审阅性。

最终目标：

- 自动理解需求或 PRD
- 自动识别需求缺失与歧义
- 自动拆分模块和流程
- 自动抽取测试点并支持人工审核
- 自动生成模块级测试用例
- 自动生成流程联动测试场景
- 自动做结构校验、覆盖校验和追溯校验

---

## 二、最终推荐总体流程

```text
平台选择 + 需求输入
  ↓
Stage 1：结构分析（全局理解）
  ↓
Stage 2：缺失检查与澄清提问（Schema + 规则驱动）
  ↓
Stage 3：模块拆分与上下文收敛（精准生成上下文）
  ↓
Stage 4：测试点提取与人工审核（质量收口）
  ↓
Stage 5：模块/功能测试用例生成（核心输出）
  ↓
Stage 6：流程联动测试生成（高级覆盖）
  ↓
Stage 7：结果校验与最终输出（结构化 + 可视化 + 可追溯）
```

---

## 三、为什么这是最终流程

这版流程融合了两套思路：

- 保留当前项目已经验证有效的主链路：`解析 -> 澄清 -> 审核测试点 -> 生成 -> 校验`
- 吸收原始方案中更强的中长期能力：`模块拆分`、`Schema 驱动缺失检查`、`流程联动测试`

设计原则：

- 当前阶段优先保证质量闭环，而不是盲目追求流程复杂度
- 长 PRD、复杂业务要能逐步拆小，而不是一次性全量生成
- 重要节点必须允许人工确认，尤其是歧义问题和测试点范围
- 模块级与流程级测试要分开建模、分开生成、最终统一输出

---

## 四、各阶段详细设计

### Stage 0：平台选择 + 需求输入

#### 目标

明确本次生成的上下文边界，为后续平台专项测试点和用例生成提供基础约束。

#### 输入

- 平台类型：`web / app / plugin`
- 原始需求文本 / PRD
- 补充信息
  - actors
  - preconditions
  - business_rules

#### 输出

- 原始输入上下文 `raw_context`

#### 说明

- 这一阶段不做生成，只负责建立工作上下文
- 平台信息会直接影响后续 `platform_focus`、测试点和测试用例

---

### Stage 1：结构分析（全局理解）

#### 目标

从需求全文中提取结构化理解结果，为后续所有阶段提供统一的中间语义层。

#### 输入

- `raw_context`

#### 输出

```json
{
  "summary": {
    "title": "用户登录",
    "business_goal": "用户输入账号密码后完成身份验证并进入首页"
  },
  "functions": ["登录", "会话建立", "首页跳转"],
  "flows": [
    "输入账号->输入密码->点击登录->校验凭证->跳转首页"
  ],
  "platform_focus": ["浏览器兼容", "会话管理", "插件权限"]
}
```

#### 关键要点

- 先形成统一摘要，再进入后续细化
- `functions` 用于模块拆分
- `flows` 用于流程联动测试
- `platform_focus` 用于平台专项测试点补充

---

### Stage 2：缺失检查与澄清提问

#### 目标

识别当前需求中影响测试设计准确度的缺失信息，并将其转化为可回答的问题。

#### 输入

- `summary`
- `functions`
- 平台类型
- 模块 schema 或平台规则模板

#### 输出

```json
{
  "missing_fields": [
    "password_policy",
    "lock_policy"
  ],
  "clarification_questions": [
    {
      "id": "CQ001",
      "question": "密码规则是什么？",
      "reason": "边界值与非法输入测试依赖该规则",
      "blocking": true
    }
  ]
}
```

#### 系统行为

1. 自动识别缺失字段
2. 自动生成澄清问题
3. 用户补充答案
4. 将答案写回结构化上下文

#### 关键要点

- 这一步是提升准确率的核心节点之一
- 建议区分 `blocking` 与 `non-blocking`
- 当前项目已实现这一思路，应该继续保留并加强

---

### Stage 3：模块拆分与上下文收敛

#### 目标

将全量需求按模块或能力域拆分，给后续测试点提取和用例生成提供更精确的局部上下文。

#### 输入

- 原始需求
- `functions`
- 用户补充的澄清答案

#### 输出

```json
{
  "module_segments": {
    "登录": "登录相关需求描述...",
    "会话建立": "登录态建立、cookie、token 相关描述...",
    "首页跳转": "登录成功后的跳转与页面反馈描述..."
  }
}
```

#### 作用

- 降低一次性传递全量 PRD 的 token 消耗
- 避免复杂需求场景下上下文混杂
- 为模块级用例生成和流程级场景生成提供不同粒度的上下文

#### 关键要点

- 这一阶段是未来升级重点
- 对 MVP 来说可以先做成「分析阶段的辅助输出」，不一定强制单独暴露给用户

---

### Stage 4：测试点提取与人工审核

#### 目标

先生成测试点，再由人工做审核和收口，避免直接生成大量不可控用例。

#### 输入

- `summary`
- `platform_focus`
- `module_segments`
- `clarification_answers`

#### 输出

```json
{
  "test_points": [
    {
      "id": "TP001",
      "title": "正确账号密码登录成功",
      "category": "正向流程",
      "description": "验证正常登录成功并进入首页",
      "source": "主流程",
      "risk_level": "high",
      "platform_specific": false
    }
  ],
  "review_notes": []
}
```

#### 人工审核动作

- 删除无效测试点
- 补充遗漏测试点
- 调整优先级和风险等级
- 控制本次生成范围

#### 关键要点

- 这是当前项目最有价值的阶段之一，必须保留
- 测试点审核是人机协作的质量收口点
- 未来可以进一步加入筛选、分组、批量审核和风险排序

---

### Stage 5：模块/功能测试用例生成

#### 目标

基于审核后的测试点生成可执行、结构化、可追溯的功能测试用例。

#### 输入

- `summary`
- `selected_test_points`
- 平台类型

#### 输出

```json
{
  "cases": [
    {
      "title": "正常登录",
      "preconditions": ["用户已注册"],
      "steps": ["输入正确账号密码", "点击登录"],
      "expected_results": ["登录成功并跳转首页"]
    }
  ]
}
```

#### 关键要点

- 只基于审核后的测试点生成，不再直接读取原始 PRD
- 生成结果必须能追溯到测试点和需求摘要
- 平台标签必须进入 `coverage_tags`

---

### Stage 6：流程联动测试生成

#### 目标

在模块级测试用例之外，补充跨模块、跨状态、跨阶段的联动测试场景。

#### 输入

- `flows`
- `summary`
- 审核后的测试点
- 模块用例生成结果

#### 输出

```json
{
  "integration_tests": [
    "登录后进入首页再执行插件授权",
    "未登录直接执行插件功能",
    "授权失败后重新登录并再次发起授权"
  ]
}
```

#### 适用场景

- 登录后再下单
- 支付失败后订单回滚
- 未授权直接访问功能
- 插件安装、授权、执行、卸载等跨阶段流程

#### 关键要点

- 当前项目尚未完整实现这一阶段
- 这是下一阶段最值得补的能力之一
- 建议作为高级覆盖补充单独输出，而不是混入普通功能用例

---

### Stage 7：结果校验与最终输出

#### 目标

对生成结果进行自动校验，最终产出可用于界面展示、导出和后续审批的结构化结果。

#### 输入

- `cases`
- `integration_tests`
- `selected_test_points`
- `summary`

#### 输出

```json
{
  "summary": {},
  "functions": [],
  "flows": [],
  "test_points": [],
  "cases": [],
  "integration_tests": [],
  "validation_issues": [],
  "traceability": []
}
```

#### 校验内容

- 字段完整性校验
- 步骤与预期结果完整性
- 平台标签校验
- 覆盖缺口校验
- 重复用例校验
- 需求/测试点/用例追溯校验

#### 关键要点

- 当前项目已经有一部分自动校验能力
- 后续建议增加 traceability matrix 输出

## 五、当前项目建议采用的落地顺序

### 第一阶段：继续沿用当前已落地链路

```text
输入需求
  ↓
解析需求
  ↓
澄清问题
  ↓
审核测试点
  ↓
生成用例
  ↓
自动校验
```

### 第二阶段：补充两个最有价值的增强点

1. 在分析阶段增加 `functions / flows / module_segments`
2. 在生成阶段后增加 `integration_tests`

### 第三阶段：升级为完整 Agent 流程

- 模块 schema 驱动缺失检查
- 模块级上下文拆分
- 流程级联动测试独立生成
- 统一 traceability 输出

## 七、核心设计原则

### 1. 结构优先

先做结构化理解，再做测试设计，不允许模型直接黑盒输出最终结果。

### 2. 人机协作优先

歧义问题和测试点审核必须允许人工参与，这是保证准确率的核心。

### 3. 模块级与流程级分层

模块级用例解决单点覆盖，流程联动测试解决端到端覆盖。

### 4. 上下文逐步收敛

通过结构分析、澄清答案和模块拆分，逐步让后续生成上下文更干净、更精确。

### 5. 全程可追溯

测试点和测试用例必须能回溯到需求摘要、模块和流程。

### 6. 先 MVP，后 Agent

先把现有链路做稳，再逐步引入更复杂的多阶段编排能力。

## 八、推荐实现映射

### 当前项目已具备

- 平台选择
- 需求输入
- 结构化解析
- 澄清问题
- 测试点审核
- 测试用例生成
- 自动校验

### 当前项目下一步建议补充

- `functions`
- `flows`
- `module_segments`
- `integration_tests`
- `traceability matrix`

### 结论

最终推荐路线不是替换当前项目流程，而是：

**以当前项目流程为主干，以 flow.md 的模块拆分与流程联动能力为增强层，逐步演进成完整的测试用例生成 Agent。**
